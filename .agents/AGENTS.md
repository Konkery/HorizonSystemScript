# Project Rules & Manifest (Horizon Automation System Scripts)

This file contains the consolidated rules, workflow standards, and guidelines imported from `GEMINI.md` and `chat-deployscript.md` for this project.

## 1. Mission & Philosophy
Этот репозиторий содержит системные скрипты и модули для развертывания промышленной платформы **Horizon Automation** на аппаратных хабах (Raspberry Pi CM4/CM5) под управлением ОС Linux.
- **Принципы:** Инфраструктура как код (IaC), Идемпотентность, Безопасность (динамическая генерация паролей).
- **Стек:** Node.js, Node-Red, MongoDB, Redis, Graylog, Grafana.
- **Архитектура:** ПО работает на 3-х уровнях (НУ — нижний, СУ — средний, ВУ — верхний уровень), каждый на отдельном хабе. Подключение к ним осуществляется по SSH с настроенными ключами (см. `.ssh/config`).

## 2. Project Hierarchy
- `Common/` (или `01__Common/`): Ядро, загрузчик конфигурации (`config-loader.js`), общие утилиты.
- `LinuxScript/` (или `02__LinuxScript/`): Настройка ОС, генерация `systemd` сервисов.
- `MongoDBscript/` (или `03__MongoDBscript/`): Инициализация СУБД, управление БД/пользователями/коллекциями.
- `NodeRedScript/` (или `04__NodeRedScript/`): Настройка среды Node-Red, аутентификация, синхронизация.

## 3. Metadata & Versioning Protocol
Все исполняемые файлы (`js`) и документы (`md`) должны содержать блок метаданных в начале файла (или в стартовом блоке комментариев для кода):
```html
<div align="left">
  <b>ООО "МАС"</b><br />
  <b>Проект:</b> 27/2025-N1-МАС<br />
  <b>Ревизия:</b> rev.XX<br />
  <b>Версия:</b> vXX<br />
  <b>Дата:</b> ДД месяц ГГГГ г.
</div>
```
- **Имена файлов:** Не должны содержать индексов ревизий/версий.
- **Правила инкремента:** 
  - При любом изменении файла (даже минимальном) версия увеличивается: `vXX + 1`.
  - При значительных изменениях логики работы модуля или структуры ревизия увеличивается: `rev.XX + 1`, при этом версия обнуляется до `v01`.

## 4. Technical Guardrails
- **MongoDB:**
  - Использовать `authSource=admin` во всех строках подключения.
  - Идемпотентность: Скрипты должны сначала проверять состояние (наличие БД/пользователей), затем применять изменения.
  - Очистка: Модуль `03__MongoDB-CreateUserAndCollection.js` принудительно удаляет все БД, отсутствующие в `settings.json` (кроме системных).
- **Credentials:**
  - Пароли генерируются динамически через `config-loader.js`.
  - Отчеты сохраняются в `res/01__config/CREDENTIALS.md`.
  - Запрещено хранение паролей в Git.

## 5. Workflow Standards (Plan -> Act -> Validate)
Перед тем как приступать к выполнению любого прикладного промта, необходимо:
1. **Актуализировать контекст проекта:**
   - Полностью прочитать файл `chat-deployscript.md` с набором промтов.
   - Ознакомиться с файлами каталога того раздела, над которым ведется работа (`Common`, `LinuxScript`, `MongoDBscript`, `NodeRedScript`), а также со связанными файлами.
2. **Детальное планирование:** Составить план изменений и согласовать его с пользователем перед внесением изменений.
3. **Исполнение:** Выполнить последовательность шагов без прерываний после утверждения плана.
4. **Валидация и актуализация:**
   - После выполнения промта обновить соответствующую документацию:
     * `README-ConfigLoader.md` (или `README-common.md`)
     * `README-LinuxScript.md`
     * `README-MongoDBscript.md`
     * `README-NodeRedScript.md`
     * `README-Deploy.md`
     * `README-Settings.md`
   - Убедиться в соответствии метаданных в измененных файлах.
   - Проверить работоспособность (тесты, линтинг).

## 6. Deployment & Censorship Guidelines
- **Новый файл — обязательная согласованность перед публикацией:** Любой новый файл, который планируется включать в tar-архив для удаленных хабов (передаваемый через `deploy-hub.js`), обязан пройти проверку на отсутствие конфиденциальных данных и наличие корректного метаданных-блока.
- **Явное согласие:** Перед добавлением файла в массив `includeItems` в `deploy-hub.js` необходимо получить явное согласие пользователя.
