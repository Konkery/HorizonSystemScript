# ПРОЕКТ 27/2025-N1-МАС<br>

_РАЗРАБОТКА СИСТЕМНЫХ СКРИПТОВ ДЛЯ ИНИЦИАЛИЗАЦИИ КОМПОНЕНТОВ HORIZON AUTOMATION_

## System instructions

### 1 Промт

#### _исходные данные:_

Данный _*системный*_ промт регулирует общие вопросы и действия которые необходимо выполнять при корректировки любого из файлов данного проекта - `27/2025-N1-МАС`.
Краткое описание проекта.<br>
Данный проект посвящен реализации различных скриптов нацеленных на разворачивание ПО платформы `Horizon Automation` (HA), прикладного ПО и системного РО на промышленные хабы работающие на промышленных микрокомпьютерах `Raspberry Pi CM4` и `CM5` под управлением ОС Linux. <br>
Все перечисленное является АСУ (автоматизированной системой управления) промышленного вендингового аппарата.<br>
Данный проект `HorizonSystemScript` является подразделом глобального проекта `27/2025-N1-МАС`, в этом подразделе разрабатываются исполняемые модули, исполняемые скрипты, сценарии оболочки нацеленные на разворачивание и подготовку всех програмных компоннетов которые формируют как основу `HA` так и прикладное ПО.
Приблизительный состав стека проекта:

- `Node.js`;
- `Horizon Automation` - платформа разработанная на языке JavaScript;
- `Node-Red` - потоковая система программирования на базе которой работает `Horizon Automation, а также пишется прикладной код;
- `MongoDB` - СУБД платформы;
- `Redis` - кэширующая СУБД;
- `GrayLog` - главный логирующий сервер;
- `Grafana` - сервер отображение дашбордов сс метриками;

Все ПО работает на 3-х уровнях:

- НУ - нижний уровень;
- СУ - средний уровень;
- ВУ - верхний уровень;
  каждый из уровней работают на отдельном аппаратном хабе. К каждому хабу можно подключится по SSH с настроенными ключами и алиасами, все это законфигурировано в файле `.ssh\config`
  Например для подключения под учетной записью:
- `operator5` к хабу НУ можно командой - `ssh hub-low-stend-op5`

Остальные записи можно узнать из файла `.ssh\config`.
Текущий список разделов над которыми ведется работа в субпроекте (т.е. текущим проекте) - `HorizonSystemScript`:

- `Common` - связующий раздел;
- `LinuxScript` - создание инстансов служб `Node-Red`, управлением питанием и т.п.;
- `MongoDBscript` - настройка и конфигурирование СУБД MongoDB;
- `NodeRedScript` - настройка и конфигурирование СУБД Node-Red;

### 5 Промт

#### _исходные данные:_

На текущем этапе развития проекта выявились серьезные в структуре файла основной конфигурации - `settings.json`.
Недостатки носят принципиальный характер. Далее я постепенно перейду к описанию структуры которую должен поддерживать файл `settings.json`.<br>
Предпосылки:

- в каждом хабе в котором есть платформа `Horizon Automation` нужно создавать основную БД - `general`, которая обеспечивает работу фреймворка `Horizon Automation`.
- каждая такая база содержит типовой набор коллекций - `collectionGeneral_HA`.
- в рамках вендингового аппарата нужно создать одну общесистемную БД - `system`.
- пользователь должен определить один конкретный хаб на котором будет развернута данная БД. А задача ПО которая будет валидировать `settings.json` убедится что в конфигурации указан ровно один хаб в котором конфигурация данной БД присутствует.
- каждая `system` БД содержит свой набор коллекций - `collectionSystem_HA`.
- к данной БД должен иметь доступ только один пользователь `system` с правами администратора.
- помимо БД которые непосредственно нужны для работы фреймворка `HA` (сюда относятся `general` и `system`) есть внешние по отношению к ядру фреймворка `HA` компоненты но которые также критически нужны для его функционирования, к ним относятся сервер потокового программирования `Node-Red`, сервер логирования `Graylog` и др. Для этих сервисов нужно хранить как как переменные конфигурации так и создавать для них БД. На данный момент с `settings.json` создан раздел - `externalServices` именно; в нем будет хранится информация как о конфигурационных переменных так и о БД.
- данные о БД внешних сервисах будет хранится в соответствующем разделе `dataBase` который вложен в уровень `externalServices`. Если `dataBase` раздел пустой (ключ "description" - не учитывать) то на данном данном хабе не требуется такая настройка. И соответственно скриптам из раздела проекта `03__MongoDBscript` не требуется работать с ними.
- данные о конфигурации других сервисов хранятся в разделе `settingsServices` который также вложен в уровень `externalServices`. На текущий момент там хранится конфигурация только одного сервиса а именно сервера потокового программирования под управлением которого работает фреймворк `HA` - `NodeRed`. Отметим что данный сервис не требует никаких дополнительных БД а также его конфигурационные файла модифицируются скриптами из модуля проекта `04__NodeRedScript`.

Далее последует описание общей картины того как нужно пользоваться файлом конфигурации проекта `settings.json`:<br>

- файл начинается с того что определяет идентификатор проекта в рамках которого он действителен - `projectID`;
- далее он определяет идентификатор вендингового аппарата в отношении которого он действителен. Для каждого экземпляра вендингового аппарата нужно создать свой файл конфигурации `settings.json`;
- далее идут секции которые определяют конфигурации самых крупных узлов ПТК _`HORIZONvending`_ (программно-технического комплекса), а именно промышленных хабов работающих под управлением платформы `HA`, комплект этих хабов и ПО на нем работающих и составляют АСУ (автоматизированную систему управления) вендингового аппарата. Секции называются:
  - `hubLow`
  - `hubMid`
  - `hubHigh`
- каждая секция хаба определяет:
  - `sessionGeneration` - флаг определяющий необходимость генерации скриптами данного проекта специфических системных и прикладных данных, нужных для работы данного хаба. Если флаг равен - `false` то скрипту валидирующего данный `settings.json` можно пропустить конфигурацию данного хаба. В случае если генерация необходима требуется то необходима полная проверка данного раздела и генерация данных, после чего флагу необходимо присвоить значение `false`, чтобы предотвратить автоматическую повторную генерацию данных для данного хаба при следующем запуске скриптов;
  - `system` - системная конфигурация хаба:
    - `hubIP` - IP-адрес хаба;
    - `hostNamePrefix` - префикс имени хоста, которое используется скриптами для получения и знания полного имени хаба, полное имя хаба получается путем конкатенации значения этого поля и значения поля `machineNAME`, порядок конкатенации:<br>
      `hostNamePrefix` + `machineNAME`;
    - `users` - конфигурация пользователей хаба, содержит массив системных пользователей хаба (по сути ОС Linux), а также пользователей которые будут иметь доступ к прикладному коду платформы 'HA', через доступ к серверу 'Node-Red' ( это очень важное замечание):
      - `userTemplate` - базовая часть имени пользователя в системе для динамического типа пользователя ('typeNameUsers' = 'dynamic') и полное имя для статического имени пользователя ('typeNameUsers' = 'static'). Для динамических имен, полное имя при генерации должно получаться путем конкатенации значения этого поля и счетчика, значение которого это положительные числа с автоинкрементом с шагом '1' и начальным значением '1' до 'countUsers';
      - `typeNameUsers` - тип имени пользователя, может иметь только два значения:
        - `static` - статический пользователь,
        - `dynamic` - динамический пользователь,<br>
          такие значения означают следующее поведение при генерации - статический пользователь создается в количестве одного экземпляра на данном хабе, динамический пользователь создается столько раз сколько указано в поле 'countUsers'. При этом ключ 'typeNameUsers' имеет более высокий приоритет чем ключ 'countUsers', т.е. если пользователь статический то значение 'countUsers' фактически не имеет влияния, так как такой пользователь однозначно может быть создан только в одном экземпляре. Если тип пользователя динамический то значение 'countUsers' обязательно должно быть указано и может находится в пределах от 1 до 10. Важное замечание - динамические пользователи создаются одновременно и для системных учетных пользователей и для конфигурирования (логин/пароль) доступа к серверу 'NodeRed', по сути это одни и теже пользователи. В отличие от СУБД в каждом сервере `Node-Red` необходимо создать всех пользователей ! Например если у нас есть 5(пять) инстансов и соответственно 5 (пять) пользователей, то в каждом из инстансов нужно настроить всех 5 пользователей и так в каждом инстансе. При этом необходимо помнить что каждый инстанс работает только в своей, отдельной учетной записи Linux, и чтобы настроить пользователей необходимо подключаться (например по SSH) к Hub под каждой из учетной записи. При этом часть команд должны выполняться с привилегией `sudo`.<br>

При формировании хэш кодов к паролям пользователей необходимо помнить что у новых версий сервера `Node-Red` поменялся подход к генерации хэш кодов паролей это необходимо учесть.
Сейчас мы работаем с таким стеком:

- `Node-RED v4.1.2`
- `Node.js v22.22.3`
- `Linux 6.12.47+rpt-rpi-v8 arm64 LE`

На компьютеров разработчиков настроена работа с `Hub` посредством SSH на основе ключей, причем ко всем системным учетным записям на Linux, по количеству равным количеству инстансов.
