## MongoDBscript

### 1 Промт

#### _исходные данные:_

Приступаем к проработке модулей предназначенных для настройки и работы СУБД MongoDB:

1. 01\_\_MongoDB-CreateUserAdmin-rev.01-v01.js
2. 02\_\_MongoDB-InitParam-rev.02-v04.js
3. 03\_\_MongoDB-CreateUserAndCollection-rev.01-v05.js

Вот назначение модулей:

- `01\_\_MongoDB-CreateUserAdmin-rev.01-v01.js` - создает `суперпользователя` в базе `admin` для дальнейшего управления СУБД;

- `02\_\_MongoDB-InitParam-rev.02-v04.js` - включает механизм авторизации в MongoDB, разрешает подключение к СУБД с удаленных хостов;

- `03\_\_MongoDB-CreateUserAndCollection-rev.01-v05.js` - выполняет полную инициализацию базы данных, коллекции и права доступа для операторов и системных служб.

Краткая справка по назначению БД, операторов, коллекций и т.д.:

- базы данных в первую очередь предназначены обслуживать платформу **Horizon Automation**, во вторую очередь предоставлять мощности СУБД для сторонних компонентов **Horizon Automation**, таких как профессиональная система логирования `Graylog` и др.;
- для одного инстанса **Horizon Automation** требуется одна БД (например `dbFramework1`), со следующей группой коллекций:
  - `rawData` - хранит "сырые", не обработанные данные, поступающие от измерительных каналов **Horizon Automation**, тип **TimeSeries**;
  - `workData` - хранит обработанные данные, поступающие от измерительных каналов **Horizon Automation**, тип **TimeSeries**;
  - `archiveData` - хранит архивные данные которые генерирует платформа **Horizon Automation**, тип **Capped**;
  - `loggingData` - хранит данные логирования которые генерирует платформа **Horizon Automation**, тип **Capped**;
  - `systemData` - хранит системные данные проекта, тип **Standart**;
  - `projectData` - хранит все конфигурационные данные необходимые для старта платформы **Horizon Automation**, тип **Standart**;
  - `devicesHandbook` - хранит "библиотеку" данных о источников и потребителей, а именно типы сенсоров, актуаторов, интеллектуальных сенсоров и т.п., тип **Standart**;

Для образовательных проектов мы создавали 5 (пять) инстансов платформы **Horizon Automation** и соответственно требовалось создавать 5 (пять) БД:

- `dbFramework1` - первый инстанс **Horizon Automation**;
- `dbFramework2` - второй инстанс **Horizon Automation**;
- `dbFramework3` - третий инстанс **Horizon Automation**;
- `dbFramework4` - четвертый инстанс **Horizon Automation**;
- `dbFramework5` - пятый инстанс **Horizon Automation**;

В каждой из которых существовал свой набор коллекций описанный выше, а также набор пользователей, также 5 (пять):

- `operator1` - для БД `dbFramework1`;
- `operator2` - для БД `dbFramework2`;
- `operator3` - для БД `dbFramework3`;
- `operator4` - для БД `dbFramework4`;
- `operator5` - для БД `dbFramework5`;

Другие БД:

- `dbGraylog` - БД обслуживает Graylog;
- `dbSystem` - БД зарезервирована для дальнейших проектов;

Коллекции для сторонних сервисов которые используются платформой **Horizon Automation** таких как `Graylog`,`Grafana`, создаются непосредственно данными сервисами, данные скрипты для них создают для них только пустую БД и пользователя.

Необходимо отметить что платформа **Horizon Automation** выполняется в тесном взаимодействии со средой потокового программирования **Node-Red**, фактически, хоть **Horizon Automation** самостоятельная платформа, но ее запуск и дальнейшая разработка прикладного ПО выполняется в **Node-Red**, поэтому в среде **Node-Red** также создает 5 (пять) пользователей, с точно такими же наименованиями как и в **MongoDB**, а также с такими же паролями, что делает все компоненты **Horizon Automation** изотропными.

#### _задача:_

То что было сейчас описано это то как есть сейчас. но нам требуется изменить как набор коллекций так и их параметры.
Первое что необходимо сделать это разработать `settings.json` файл (или предложить другой рациональный способ) для хранения такой информации как:

- `имя пользователя` - логин пользователя в экосистеме **Horizon Automation**;
- `пароль пользователя` - пароль пользователя в экосистеме **Horizon Automation**;
- `название БД` - в СУБД **MongoDB**;
- `наименование коллекции` - в СУБД **MongoDB**;
- `тип коллекции` - в СУБД **MongoDB**;
- `параметры коллекции` - размер ограничений, если данная коллекция типа **TimeSeries**, **Capped** и `none` для коллекций типа **Standart**;
- `номер порта службы nodered` - порт который будет назначен для работы службы аналогичной `noderedXXX.service`.

Нужна такая система предварительной конфигурации и соответствующая поддержка на стороне модулей `js` что бы ПО в модуле само понимало сколько нужно инстансов подготовить, причем это касается как системных скриптов которые находятся в директории `LinuxScript`, таких как `noderedXXX.service`. Так и скриптов находящихся в директории `MongoDBscript`(предназначенных для работы с MongoDB), а также и скриптов находящихся в директории`NodeRedScript` (предназначенных для работы с Node-Red).

Таким образом необходимо разработать новый скрипт `js` который будет генерировать (создавать) такие системные скрипты как `noderedXXX.service`, назначая порты начиная с `min` значения и увеличивая для следующих инстансов номер порта на 1. Сейчас пять скриптов созданы изначально и сразу выложены в директорию `\LinuxScript\NodeRed` а после модификации требуется чтобы появился новый `js` модуль который будет запускаться и создавать файлы с такими службами в директорию `\LinuxScript\NodeRed` где он сам также должен находится. При новом запуске, модуль должен удалить результаты старой генерации, по сути в данной директории после старта модуля `js` он должен удалить все файлы которые не являются частью исполняемых модулей `js`

Далее все модули из директории `MongoDBscript` также должны быть модифицированы на использования при старте файла конфигурации `settings.json`, читая требующиеся для их работы данные.
При старте они должны проводить валидацию `settings.json` (возможно под это необходимо сделать отдельный `js` модуль который будет оказывать "услугу" другим модулям по валидации), считывать необходимые данные и динамически создавать столько БД, пользователей и т.п. сколько будет указано в файле конфигурации `settings.json`.

#### _задача:_

Что имелось ввиду, конфигуратор должен содержать примерно следующую информацию:

- количество инстансов, возможные варианты от минимальное значение равно `1`максимальное `5`, с шагом `1`;
- стартовый порт для сервисов `nodered`= 2001, для инстансов №2 и далее, номер порта в самом модуле `js` должен увеличиваться на `1`
- не нужно непосредственно перечислять конкретных пользователей, модули `js` должны взять базовое имя - `operator`, а далее модуль должен, зная сколько инстансов нужно 'поднять', добавлять в конец имени текущий индекс от `1` и далее (с шагом `1`) формировать конечное имя пользователя, таким образом если необходимо поднять `3` инстанса то модуль сгенерирует 3 имени - `operator1`, `operator2`, `operator3`.
- паролей пользователей не должны присутствовать в `settings.json`, пароли необходимо сгенерировать 'налету' при исполнении `js` модуля, при этом эти пароли необходимо сохранить в некоторый результирующий файл, текстового формате (жду предложений по типу файла), из которого они будут взяты всеми остальными исполнительными модулями и скриптами.<br><br>

Пароль необходимо формировать в соответствующими правилами:

- `длина пароля, символов`: 8;
- `состав пароля`: при генерации использовать малые и строчные символы, цифры, специальные символы;
- `срок жизни пароля`: уничтожаться при каждом запуске моделей `js`.<br>

Имена БД СУБД `MongoDB` также не нужно дословно перечислять, достаточно иметь также 'базовую' часть имени- `dbHA`, далее также как и с паролями необходимо добавлять в конеце имени БД цифровой префикс `1` с шагом 1, т.е. при 3-х инстансов. имена БД получатся такими - `dbHA1`, `dbHA2`, `dbHA3`.
При этом есть важный момент который был упущен, а именно, каждому имени БД должен предшествовать префикс - 'имя программно технического комплекса автоматизированной системы управления вендинговым аппаратом' отделяемый символом подчеркивания:

- `VendingMachineName`: **MAS-n1701-F1757N001-VMhyb-01_dbHA1**

Таким образом общее имя БД должно получится следующим:

- `dbName`: **MAS-n1701-F1757N001-VMhyb-01_dbHA1**

При этом значение ключа `VendingMachineName` должен определять пользователь, а конечное имя БД должно формироваться динамически в момент работы исполняемого модуля `js` в зависимости от инстанса.<br>

Вышеописанное правило формирование имени БД, относится только к БД которые непосредственно использует платформа `Horizon Automation`. другие БД, например та что использует `Graylog`, как и пользователи таких БД, должны задаваться статически в файле конфигурации `settings.json`. При этом пароли также должны генерироваться динамически и также сохраняться во внешнем файле.<br>

Нужно внести изменения в наименование коллекций, необходимо переименовать коллекцию `workData`:

- старое наименование: `workData`
- новое наименование: `fineData`

Для коллекций также не нужно создавать статическую конфигурацию заранее на N инстансов. Необходимо задать конфигурацию для одного, базового, набора коллекций и "клонировать" данные характеристики для всех инстансов.<br>

Еще раз акцентирую внимание на том что такие параметры как:

должны задаваться пользователем в файле `settings.json`, и так как это сменные параметры то конкретное значение такого ограничения не нужно описывать в таких файлов как `README-MongoDBscript.md`, комментировать что коллекция имеет такой то тип ограничения нужно а конкретное значение не нужно.<br>
