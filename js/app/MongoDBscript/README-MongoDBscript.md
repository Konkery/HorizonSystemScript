# MongoDBscript: Динамическое управление СУБД Horizon Automation

Директория содержит набор инструментов для автоматизированного развертывания и управления СУБД MongoDB. Основная особенность текущей версии — полная зависимость от конфигурационного файла `res/settings.json`, что позволяет масштабировать систему без правки исходного кода скриптов.

## Реестр баз данных и динамическое именование

Система Horizon Automation использует иерархическую структуру имен для обеспечения уникальности данных в рамках парка вендинговых аппаратов.

### 1. Формирование имени БД
Имя каждой базы данных формируется динамически при старте скриптов:
`[Имя_Аппарата]_[Базовое_Имя_БД][Номер_Инстанса]`

Пример при `machineName: "MAS-n1701-F1757N001-VMhyb-01"`:
*   **MAS-n1701-F1757N001-VMhyb-01_dbHA1** — база данных для первого инстанса.

> [!IMPORTANT]
> **Внимание к символам:** В учетных системах имя ПТК может записываться через точку (например, `...VMhyb.01`). Однако **MongoDB не рекомендует использовать точки в именах баз данных**. 
> При заполнении `settings.json` обязательно заменяйте точки на дефисы (используйте `-01` вместо `.01`), чтобы избежать ошибок в работе драйверов и систем логирования.

### 2. Безопасность и пароли
Пароли для всех пользователей (`operatorN`, `graylog`, `system`) больше не хранятся в открытом виде в `settings.json`.
*   **Генерация:** Пароли создаются автоматически (8 символов: буквы, цифры, спецсимволы).
*   **Хранение:** Сгенерированные учетные данные сохраняются в файл `res/generated_secrets.json`. 
*   **Срок жизни:** Пароли обновляются при каждом новом цикле генерации системных служб.

## Структура и типы коллекций

Для каждого инстанса создается набор коллекций. Конкретные параметры (размер, TTL) настраиваются в `res/settings.json`.

| Коллекция | Тип | Назначение | Параметры / Ограничения |
| :--- | :--- | :--- | :--- |
| `rawData` | **TimeSeries** | Хранение "сырых" данных с датчиков. | Ограничение по TTL (времени) |
| `fineData` | **TimeSeries** | (Ранее `workData`). Обработанные данные. | Ограничение по TTL (времени) |
| `archiveData` | **Capped** | Архив важных событий и алармов. | Ограничение по объему (МБ) |
| `loggingData` | **Capped** | Внутренние логи работы платформы. | Ограничение по объему (МБ) |
| `systemData` | **Standart** | Системные флаги и состояния. | Без ограничений |
| `projectData` | **Standart** | Конфигурация проекта. | Без ограничений |
| `devicesHandbook` | **Standart** | Справочник типов устройств. | Без ограничений |

## Порядок выполнения скриптов

Все модули автоматически считывают параметры подключения и учетные данные из `settings.json` и проводят валидацию конфигурации перед стартом.

1.  **`01__MongoDB-CreateUserAdmin-rev.03-v01.js`**
    *   Создает суперпользователя `admin` с ролью `root`. Используется один раз при развертывании новой СУБД.
2.  **`02__MongoDB-InitParam-rev.03-v01.js`**
    *   Настраивает `/etc/mongod.conf`. Разрешает внешние подключения и активирует авторизацию. **Требует перезапуска службы.**
3.  **`03__MongoDB-CreateUserAndCollection-rev.03-v01.js`**
    *   **Ключевой скрипт.** Динамически создает столько БД и пользователей, сколько указано в `settings.json`.
    *   Автоматически применяет настройки `TimeSeries` и `Capped` для соответствующих коллекций.
    *   ВНИМАНИЕ: Скрипт удаляет существующие данные в целевых БД перед пересозданием.

## Техническое примечание для разработчиков

Для работы скриптов используется общий модуль валидации `js/app/Common/config-loader-rev.03-v01.js`. При добавлении новых типов коллекций или изменении структуры данных достаточно внести правки в `res/settings.json`.
