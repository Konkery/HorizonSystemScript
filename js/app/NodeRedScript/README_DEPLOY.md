# Horizon Hub: Система автоматизации настройки Node-RED

## 1. Постановка задачи
Необходимо обеспечить централизованную настройку безопасности для промышленного Хаба "Horizon Hub" (Raspberry Pi CM4/5), на котором запущен фреймворк "Horizon Server".

### Параметры системы:
*   **Инстансы:** 5 независимых копий Node-RED, работающих под системными пользователями `operator1`, `operator2`, `operator3`, `operator4`, `operator5`.
*   **Порты:** Инстансы распределены по TCP-портам: 2001 (`operator1`) ... 2005 (`operator5`).
*   **Пользователи в каждом инстансе:** Каждый инстанс Node-RED настроен на поддержку **всех пяти** операторов (`operator1`-`operator5`) и системного пользователя (`system`).
*   **Источник данных:** Файл конфигурации MongoDB `js/app/MongoDBscript/03__MongoDB-CreateUserAndCollection-rev.03-v01.js`, содержащий массив паролей `dbUser`.
*   **Цель:** Автоматическая синхронизация паролей из файла MongoDB в файлы `settings.js` всех инстансов Node-RED.

---

## 2. Ключевые проблемы и их решение

### Проблема А: Несовместимость алгоритмов хеширования
**Суть:** Современные библиотеки Node.js (v24+) на Windows генерируют bcrypt-хеши с префиксом `$2b$`. Версия Node-RED на Хабе (v4.1.2) ожидает хеши с префиксом `$2y$`, созданные системной утилитой.
**Решение:** Хеширование выполняется **непосредственно на Хабе** путем вызова команды `node-red admin hash-pw` внутри SSH-сессии для каждого пользователя.

### Проблема Б: Права доступа (Permissions)
**Суть:** Если создавать файл конфигурации через `sudo` или из-под другого пользователя, Node-RED не сможет сохранить изменения в потоках (Flows) из-за отсутствия прав записи.
**Решение:** Скрипт использует **персональные SSH-сессии** для каждого оператора. Файл `settings.js` создается от имени самого пользователя, что автоматически делает его владельцем файла.

### Проблема В: Кроссплатформенность (Windows -> Linux)
**Суть:** Терминалы Windows (PowerShell/CMD) не поддерживают синтаксис `Heredoc` (`<<EOF`).
**Решение:** Скрипт на ПК инженера динамически создает временный `.sh` файл, передает его на Хаб через `scp` и исполняет через `bash`.

## 3. Инструментарий

### 1. `setup_master_keys-rev.03-v01.js` (Подготовка)
Используется один раз для нового Хаба. Рассылает публичный SSH-ключ инженера во все учетные записи Хаба. Это позволяет проводить деплой без постоянного ввода паролей.

### 2. `master_deploy-rev.03-v01.js` (Основной инструмент)
Главный скрипт пусконаладки. 
*   **Локально:** Парсит JS-файл MongoDB для извлечения паролей всех операторов.
*   **Дистанционно:** Итерирует пользователей, заходит в каждый аккаунт, генерирует локальные хеши для всего списка операторов, обновляет `settings.js` и перезапускает службу через Мастер-аккаунт (`operator3`) с правами `sudo`.

---

## 4. Эксплуатация (Инструкция)

### Первичная настройка (для нового Хаба):
1.  Сгенерировать ключ на ПК: `ssh-keygen`.
2.  Настроить доступ к `operator3`: `ssh-copy-id operator3@<IP>`.
3.  Разослать ключи остальным: `node js/app/NodeRedScript/setup_master_keys-rev.03-v01.js`.

### Синхронизация паролей:
При изменении паролей в `03__MongoDB-CreateUserAndCollection-rev.03-v01.js`:
1.  Убедиться, что файл сохранен (SFTP залил его на Хаб).
2.  Запустить деплойер с ПК:
    ```bash
    node js/app/NodeRedScript/master_deploy-rev.03-v01.js
    ```

## 5. Текущий статус (v4.4)
На данном этапе автоматизация настроена для инстансов **operator1, operator2, operator3**. 
В каждом из этих инстансов созданы учетные записи для всех 5 операторов.
Инстансы **operator4** и **operator5** временно исключены из цикла обновления до момента подтверждения их SSH-доступа.
