# README-Settings: Детальное описание конфигурации `settings_ha.json`

Этот документ содержит исчерпывающее описание структуры файла `settings_ha.json`, используемого для настройки АСУ вендингового аппарата.

## Общий вид структуры конфигурации

Для понимания иерархии приводится пример структуры с полностью раскрытым разделом `hubMid`:

```json
{
  "projectID": "27/2025-N1-МАС",
  "machineNAME": "MAS-n1701-F1757N001-VMhyb-01",
  "description": "Конфигурация трёхуровневой АСУ вендингового аппарата",
  "globalUsers": [{ "userTemplate": "system", "typeNameUsers": "static", "role": "administrator" }],
  "hubLow": { ... },
  "hubMid": {
    "description": "Конфигурация АСУ - средний уровень",
    "sessionGeneration": false,
    "system": {
      "hubIP": "10.130.1.12",
      "hostNamePrefix": "hubMid",
      "users": [ { "userTemplate": "operator", "typeNameUsers": "dynamic", "role": "operator", "countUsers": 1 } ]
    },
    "internalServices": {
      "description": "Конфигурация программных компонентов фреймворка HA - средний уровень",
      "generalDatabase": {
        "description": "Конфигурация рабочих БД фреймворка HA",
        "horizonAutomation": {
          "typeDBMS": "MongoDB",
          "dbPostfix": "-general-ASUmid-HA",
          "collectionGeneral_HA": {
            "description": "Шаблоны коллекций для основной БД фреймворка HA",
            "list": [ { "name": "fineData", "type": "TimeSeries", "params": { ... } } ]
          }
        }
      },
      "systemDatabases": {
        "description": "Конфигурация системных БД вендингового аппарата",
        "systemVM": {
          "typeDBMS": "MongoDB",
          "dbPostfix": "-system-VM",
          "collectionSystem_HA": { ... }
        }
      }
    },
    "externalServices": { ... }
  },
  "hubHi": { ... }
}
```

---

## 1. Корневой раздел (Глобальная конфигурация)

Определяет контекст функционирования всего аппарата.

| Ключ | Тип | Описание |
| :--- | :--- | :--- |
| `projectID` | String | Идентификатор проекта (обязателен). |
| `machineNAME` | String | Уникальное имя аппарата. Используется как базовый префикс для формирования имен всех баз данных (например, `machineNAME_dbHA1`). |
| `description` | String | Краткое описание конфигурации. |
| `globalUsers` | Array | Список глобальных пользователей, присутствующих на всех хабах (например, системный администратор `system`). |

---

## 2. Раздел Хаба (Структура секции хаба)

Каждый из хабов (`hubLow`, `hubMid`, `hubHi`) имеет изотропную структуру. Ниже приведено описание каждого ключа в рамках секции хаба.

### 2.1. Системные настройки (`system`)
| Ключ | Тип | Описание |
| :--- | :--- | :--- |
| `hubIP` | String | IP-адрес для подключения к хабу. |
| `hostNamePrefix` | String | Префикс для генерации полного имени хоста (конкатенируется с `machineNAME`). |
| `users` | Array | Массив локальных пользователей хаба (операторы). |

#### Подраздел `users` (внутри `system`)
| Ключ | Тип | Описание |
| :--- | :--- | :--- |
| `userTemplate` | String | Базовое имя пользователя (например, `operator`). |
| `typeNameUsers` | String | Тип: `static` (один экземпляр) или `dynamic` (автоматическая генерация с суффиксом). |
| `role` | String | Роль (например, `operator`, `administrator`). |
| `countUsers` | Number | Количество инстансов (только для `dynamic`). |

### 2.2. Внутренние сервисы (`internalServices`)
| Ключ | Тип | Описание |
| :--- | :--- | :--- |
| `generalDatabase` | Object | Настройка рабочих БД (Horizon Automation). |
| `dbPostfix` | String | Суффикс для уникального имени БД (формирует `machineNAME` + `dbPostfix`). |
| `collectionGeneral_HA`| Object | Описание шаблонов коллекций. Каждая коллекция имеет поля `name`, `description`, `type` (`TimeSeries`, `Capped`, `Standart`), и `params` (параметры для настройки, включая `granularity`). |
| `systemDatabases` | Object | Системные БД (например, `systemVM`). |

### 2.3. Внешние сервисы (`externalServices`)
| Ключ | Тип | Описание |
| :--- | :--- | :--- |
| `dataBase` | Object | Настройка внешних БД (например, `grayLog`). |
| `settingsServices`| Object | Переменные внешних сервисов (например, `nodeRed` с параметром `startPort`). |

---

## 3. Сценарий: Конфигурация аппарата (пример отличий)

В ПТК HORIZONvending аппарат состоит из 3-х хабов. Несмотря на изотропную структуру, наполнение различается для реализации бизнес-логики.

| Параметр | `hubLow` | `hubMid` | `hubHi` |
| :--- | :--- | :--- | :--- |
| **Инстансы операторов** | `5` (`operator1`-`5`) | `1` (`operator1`) | `1` (`operator1`) |
| **Системная БД** | пусто | сконфигурирована (`systemVM`) | пусто |
| **GrayLog** | пусто | сконфигурирован | пусто |
| **Node-RED Port** | 2001 | 2001 | 2001 |

### Пояснение по конфигурации БД:
*   **Системная БД (`systemVM`)**: Сконфигурирована **только в `hubMid`**. В `hubLow` и `hubHi` раздел присутствует для сохранения структуры, но содержимое явно помечено как `(пусто на данном хабе)`. Это предотвращает конфликтные попытки создания одной системной БД на трех разных хабах, при этом структура файла остается валидной и одинаковой для парсинга.
*   **Генерация пользователей**: Хотя структура `system -> users` одинакова везде, `countUsers` варьируется. Например, `hubLow` генерирует 5 операторов для управления большой периферией, а `hubHi` — только 1 для задач мониторинга.

---

## Правила редактирования
1.  **Принцип изотропности**: При добавлении нового ключа в один из хабов, он **должен** быть добавлен во все остальные (даже если он пуст).
2.  **Описание**: Всегда заполняйте поле `description`, чтобы будущие инженеры понимали назначение секции или причину её пустоты.

## Техническое примечание
